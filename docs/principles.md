# Principles — このロードマップの判断基準

学習・資格・プロジェクトに関する意思決定は、以下の原則に従う。
迷ったらこのファイルに立ち返る。原則自体を変更する場合は ADR に記録する。

## 1. 資格は目的ではなく、実装力を高めるための手段

- 「資格を取ること」自体をゴールにしない
- 資格を選ぶ際は [docs/certification/overview.md](certification/overview.md) の評価軸で「開発者としての実用性」を最優先する
- 実装力・設計力・キャリアに繋がらない資格は、候補にあっても取らない判断をしてよい

## 2. 学習した技術は必ず小さな成果物に落とし込む

- 「読んだ」「動画を見た」で終わらせず、必ず手を動かして小さな成果物（リポジトリ、デプロイ、記事）にする
- 資格学習も同様に、合格後に作る成果物をあらかじめ [templates/certification-plan.md](../templates/certification-plan.md) で決めておく

## 3. 公式情報を最優先する

- Microsoft 資格は必ず Microsoft Learn の公式試験ページで、試験名・言語・廃止日・後継資格・更新日・試験範囲を確認する
- CNCF / GitHub / HashiCorp なども各公式サイトを一次情報とする
- 技術仕様は公式ドキュメント・仕様書（RFC 等）を一次情報とする

## 4. 古い試験情報やブログ記事を鵜呑みにしない

- 資格・AI サービスは変更が激しい。ブログ・Qiita・書籍・AI の回答は「執筆時点の情報」として扱う
- 特に AI 系資格・Azure AI サービスは名称や構成の変更が頻繁なため、必ず公式ページで現在の状態を確認してから計画に反映する
- このリポジトリに情報を書くときも、断定できないものには「要公式確認」と明記する

## 5. AI エージェントを活用するが、最終判断は自分で行う

- 調査・要約・コード生成・レビューには AI エージェントを積極的に使う
- ただし、受験判断・キャリア判断・アーキテクチャ判断などの意思決定は自分で行い、重要なものは ADR に残す
- AI の出力は必ず検証する（特に日付・価格・試験情報などの事実情報）

## 6. 広く浅くではなく、開発者として使う技術を深く学ぶ

- 「試験に出るから」ではなく「自分のプロダクトで使うから」学ぶ
- 管理者・営業向けの範囲よりも、開発者として手を動かす範囲を深掘りする
- 使わない技術の資格を数集めるより、使う技術を Lv3（設計できる）以上に引き上げる

## 7. クラウドコスト・セキュリティ・運用性を常に考慮する

- 個人開発でもコストの見積もりと上限設定（Budget / アラート）を必ず行う
- シークレットをコードに埋め込まない。Managed Identity / Key Vault を基本とする
- 「動いた」で終わらせず、ログ・監視・障害時の対応まで含めて「運用できる」状態を目指す
