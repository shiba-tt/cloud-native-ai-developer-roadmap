# Vision — Cloud Native AI Developer

## 何を作れる人を目指すのか

「アイデアを、モバイルアプリ + クラウドバックエンド + AI 機能を含むプロダクトとして、一人で形にして運用できる人」を目指す。

具体的には、次のようなプロダクトを企画から運用まで一貫して作れること。

- SwiftUI 製の iOS アプリと、それを支える Azure 上のバックエンド API
- Cosmos DB / SQL / Vector Search を組み合わせたデータ基盤
- RAG や Agent を組み込んだ AI 機能（検索、要約、対話、自動化）
- Entra ID / OAuth / OIDC による安全な認証・認可
- コンテナ化され、IaC でプロビジョニングされ、CI/CD で継続的にデリバリーされる運用基盤
- メトリクス・ログ・トレースによる監視と、コストを意識した運用

## どのような技術領域を扱うのか

| 領域 | 内容 |
| --- | --- |
| Application | iOS (Swift / SwiftUI)、Web、バックエンド API |
| Cloud Platform | Azure（App Service / Functions / Container Apps / AKS など） |
| Data | SQL / NoSQL / Cosmos DB / Vector Search / データモデリング |
| AI | Azure AI / Azure OpenAI / RAG / Agent / Evaluation |
| Security | Entra ID / OAuth / OIDC / Managed Identity / Key Vault |
| Cloud Native | Docker / Kubernetes / IaC (Terraform, Bicep) / Observability |
| DevOps | CI/CD / GitHub Actions / テスト自動化 / リリース戦略 |
| Product | 個人開発プロダクトの企画・設計・改善 |

## なぜクラウドネイティブと AI を組み合わせるのか

- AI 機能（LLM / RAG / Agent)は単体では価値にならず、**データ・認証・API・運用**を含むアプリケーション全体として初めてプロダクトになる
- AI アプリケーションは推論コスト・レイテンシ・スケーラビリティの制約が大きく、**クラウドネイティブな設計（コンテナ、スケーリング、Observability）と切り離せない**
- 「AI を呼べる人」は増えるが、「AI を含むシステム全体を設計・運用できる人」は希少であり、キャリア上の差別化になる
- Azure は AI サービス（Azure OpenAI / Azure AI Foundry など）とクラウド基盤が統合されており、AZ-204 で得た基盤知識をそのまま AI アプリ開発に接続できる

## iOS エンジニアとしての強みをどう活かすのか

- **UI/UX とクライアント実装力**: AI 機能は最終的にユーザー体験として届く。モバイルの体験設計・実装ができることは、AI プロダクト開発における大きな強み
- **クライアント視点の API 設計**: モバイルクライアントを長く書いてきた経験を、使いやすい API・バックエンド設計に還元できる
- **エンドツーエンドの視点**: 「アプリ → API → データ → AI → 運用」を一人で貫通できる開発者は少ない。iOS を起点に守備範囲を後ろへ広げる
- **個人開発との親和性**: App Store でのリリース経験・配布ノウハウを活かし、学んだバックエンド/AI 技術をすぐプロダクトとして世に出せる

## 5 年後の理想像

- 個人開発プロダクトを複数運用しており、そのうち少なくとも 1 つは AI 機能（RAG / Agent）を中核とするプロダクトである
- モバイル / バックエンド / データ / AI / セキュリティ / 運用を横断してアーキテクチャを設計・説明できる
- Azure を軸に、Kubernetes / IaC / Observability を実務レベルで扱える
- 資格はロードマップ上の必要なものを取得済みで、その知識がすべて成果物に反映されている
- 「iOS もできる Cloud Native AI Developer」として、市場価値の高いポジション（テックリード / プロダクトエンジニア / フリーランス等）を選択できる状態にある
